# AI Development Workflow

> 본 문서는 이번 프로젝트를 진행하며 정립된 AI 협업 개발 워크플로우를 정리한다.
>
> 특정 프로젝트에 종속되지 않으며, 향후 새로운 프로젝트에서도 동일한 방식으로 활용할 수 있는 개발 프로세스를 목표로 한다.

---

# 목적

AI는 매우 빠르게 구현할 수 있지만,
잘못된 방향으로 구현하면 수정 비용도 매우 커진다.

따라서 구현 속도보다 **설계 품질**을 우선하는 개발 흐름을 유지한다.

---

# 기본 원칙

## 1. Observation → Research → Decision → Implementation

새로운 아이디어가 떠올랐다고 바로 구현하지 않는다.

항상 아래 순서를 따른다.

```
Observation
    ↓
Research
    ↓
Decision
    ↓
Implementation
    ↓
Testing
    ↓
Real Usage
    ↓
Observation...
```

---

## 2. 구현보다 문서를 먼저 수정한다

가능하면

```
문서 수정

↓

AI 구현
```

순서를 유지한다.

문서가 최신 상태라면
AI가 변경 의도를 오해할 가능성이 크게 줄어든다.

---

## 3. Decision은 충분한 검토 이후 작성한다

새로운 아이디어가 나왔다고
바로 Decision으로 승격하지 않는다.

먼저

- 장점
- 단점
- 기존 설계와의 충돌
- 영향 범위

를 충분히 검토한다.

---

## 4. Research는 자유롭게 기록한다

Research는

"좋은 아이디어"

를 적는 곳이다.

아직 구현 여부가 결정되지 않은 내용도
부담 없이 기록한다.

Research는 버리지 않는다.

몇 주 후 다시 중요한 설계가 될 수도 있다.

---

## 5. 구현과 연구를 분리한다

Research

↓

Decision

↓

Implementation

세 영역을 섞지 않는다.

현재 구현되지 않았더라도
Research에는 충분히 기록할 수 있다.

## 6. AI에게 무엇을 만들지보다 무엇을 건드리지 말아야 하는지를 알려준다.

예시 규칙 : 프리젠터 프로젝트 기준

History는 유지
Store 구조 유지
최소 수정 원칙
Breaking Change Rule 준수
기존 Decision과 충돌 검토

등의 규칙으로 AI에게 Rule을 주었을 때 구현 품질이 훨씬 좋아졌고 긴 작업을 줄 때도 일관성을 유지하는 편이됨
"목표"와 함께 "건드리면 안 되는 것"을 항상 같이 적어주면 컨텍스트가 길어지거나, 모델들이 제각각 달라져도 일관성이 생기고 이는 품질 향상과 경제적인 개발로 이어짐

---

# AI 활용 방식

## AI는 구현자가 아니라 설계 파트너다.

구현만 시키기보다

다음과 같은 질문을 먼저 한다.

- 기존 설계와 충돌하는가?
- 영향 범위는 어디까지인가?
- Store에는 어떤 영향이 있는가?
- Persistence는 수정되는가?
- Undo는 영향을 받는가?
- 기존 Decision을 폐기해야 하는가?

---

## 구현 전 영향 분석을 요청한다

예시

```
Store 영향 분석

Persistence 영향 분석

Domain 영향 분석

UI 영향 분석

Migration 영향 분석
```

이 과정을 거치면
대규모 리팩토링 위험을 크게 줄일 수 있다.

---

## AI에게 선택지를 요구한다

정답 하나를 요구하기보다

```
A안

B안

C안
```

처럼

여러 설계안을 비교하도록 요청한다.

Trade-off를 이해한 뒤 결정한다.

---

## 여러 AI의 의견을 비교한다

가능하다면

GPT

↓

Claude

↓

GPT

↓

Claude

처럼

설계를 교차 검토한다.

모든 의견을 따르기보다

공통적으로 지적되는 문제를 우선 검토한다.

## AI가 이해해야 하는 것은 코드보다 설계 판단이다.

AI는 코드를 읽는 것보다 설계 판단의 근거를 이해할 때
가장 안정적으로 작업을 이어갈 수 있다.

코드는 현재 상태를 보여주지만

문서는

왜 그렇게 구현했는가
왜 구현하지 않았는가
어떤 위험 때문에 보류했는가

를 기록한다.

장기 프로젝트에서는

코드보다 Decision Log가
컨텍스트를 복원하는 데 더 큰 도움이 된다.

---

# 실사용 중심 개발

## 실제 사용하면서 문제를 찾는다

좋은 아이디어는

회의보다

실사용에서 나온다.

예시

- Live 버튼을 다시 누르면 끌 수 있어야 한다.
- 원하는 위치에 Page를 삽입하고 싶다.
- Browser가 필요하다.
- Library가 필요하다.

이런 요구사항은
실사용 과정에서 자연스럽게 발견된다.

---

## 기능보다 Workflow를 관찰한다

"무슨 기능을 넣을까?"

보다

"사용자는 어떤 순서로 작업하는가?"

를 먼저 본다.

UI보다 Workflow가 우선이다.

---

## 실제 프로그램을 많이 사용한다

동일 분야의 프로그램을 직접 사용하면

기능보다

Workflow 차이를 이해할 수 있다.

예시

- WorshipTools
- ProPresenter

좋은 UX는
기능보다 작업 흐름에서 발견된다.

---

# TODO 관리

## 쉬운 작업보다 의존성을 본다

우선순위보다

의존 관계를 먼저 본다.

예시

```
Section Migration

↓

Drag & Drop

↓

Browser

↓

Library
```

앞의 작업 하나가

뒤의 여러 작업을 단순하게 만들 수도 있다.

---

## TODO는 구현과 연구를 함께 관리한다

구현 TODO와

Research TODO를 구분한다.

예시

Implementation

- Storage Adapter
- Overlay
- Section Migration

Research

- Browser Architecture
- Library Workflow
- Editor Separation

---

# 문서 작성 원칙
## 기본 원칙

새로운 설계가 나오면

먼저

Research

↓

충분한 검토

↓

Decision

↓

CurrentState 반영

↓

구현

순서를 유지한다.

## 설계 판단을 기록한다

문서는 기능 설명보다

설계 판단의 이유를 기록하는 데 더 큰 가치를 가진다.

리팩토링이나 설계 변경이 끝나면 가능하면 다음 내용을 함께 남긴다.

무엇을 변경했는가

무엇을 변경하지 않았는가

왜 그렇게 판단했는가

다음 작업은 무엇인가

이 기록은 시간이 지나도

프로젝트의 설계 의도를 빠르게 복원할 수 있게 해준다.

AI와 협업할 경우

새로운 세션에서도
동일한 분석을 반복하지 않고

이전 판단을 이어서 작업할 수 있다.
---

# 프로젝트가 성장할수록 중요한 것

초기에는

코드를 얼마나 빨리 작성하는지가 중요하다.

그러나 프로젝트가 커질수록

설계를 얼마나 오래 유지할 수 있는지가 더 중요하다.

문서는 구현을 설명하기 위한 것이 아니라

앞으로의 구현 방향을 합의하는 도구다.

---

# 프로젝트를 통해 얻은 교훈

- 문서는 코드보다 먼저 수정한다.
- 구현보다 설계를 먼저 검토한다.
- AI는 구현자가 아니라 설계 파트너로 활용한다.
- Research를 적극적으로 축적한다.
- 실제 사용하면서 Workflow를 개선한다.
- 좋은 UX는 기능보다 작업 흐름에서 나온다.
- 하나의 AI보다 여러 AI의 관점을 비교하는 것이 설계 품질을 높인다.
- Observation → Research → Decision → Implementation 사이클을 반복하며 프로젝트를 발전시킨다.
- 코드만큼 설계 판단의 기록도 중요한 프로젝트 자산이다.
- AI는 기억하는 것이 아니라 기록을 빠르게 이해한다.
- 좋은 문서는 기능 설명보다 설계 의도를 보존한다.
- 리팩토링은 코드 이동보다 위험 요소를 먼저 찾는 과정이다.
- 위험도가 높은 구조는 억지로 정리하기보다 의도를 문서화하는 것이 더 안전할 수 있다.
