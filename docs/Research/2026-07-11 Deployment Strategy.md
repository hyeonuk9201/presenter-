## 1. 사실 확인 (2026-07-11 기준, 코드/문서/이력 대조)

판단에 앞서 현재 배포 형태를 코드와 세션 이력에서 역산했다. 문서에
배포 구성이 명시된 곳은 없어서, 일부는 추정이다(추정에는 표시).

1. **배포 형태**: git 저장소 + 로컬 정적 서버 + 브라우저 접속. 빌드
   스텝 없음(D-028이 "index.html/output.html은 여전히 `<script
   type="module">`로 직접 로드되는 정적 파일 구조를 유지한다"고 명시).
   근거: 9-10 실사용 디버깅 기록의 "Pull 최신 반영 확인, 서버 재시작",
   Playwright 검증의 "로컬 정적 서버". `file://` 직접 열기는 ES Modules
   CORS 제약으로 불가능하므로 서버는 필수 구성 요소다.
2. ~~**주 사용 환경**: **iPad/Safari**~~ → **정정됨(2026-07-11, 사용자
   확인)**: 실제 운영 환경은 **Windows/Mac 데스크톱·노트북 중심**이다.
   초판이 근거로 삼았던 iPad 이력(CurrentState 섹션 6 "iPad/Safari
   환경(정현욱의 주 사용 환경)", 9-8 iPad 실사용 검증, 9-9/9-10 iOS
   전용 버그 수정)은 테스트/검증 이력이지 운영 표준이 아니었다 —
   코드 이력만으로 운영 환경을 역산한 것이 초판의 오류였다. iPad
   호환성(playsinline, text-shadow 합성 등)은 이미 코드에 확보된
   자산으로 남으며, 데스크톱 중심 전략과 충돌하지 않는다.
3. **데이터 위치**: 조작하는 브라우저 안. localStorage(Presentation/
   AppSettings/Song — StorageAdapter 경유) + IndexedDB(Media blob —
   MediaStore). **서버에는 데이터가 없다** — 서버는 정적 파일만 서빙하고,
   상태는 전부 클라이언트 브라우저 저장소에 있다.
4. **백업/내보내기 기능: 없음** (전역 검색으로 확인 — 내보내기/백업/
   download 관련 사용자 기능이 코드·TODO 어디에도 없다).
5. **Local First의 원래 의도와 현재 구현의 간극**: `Vocabulary.md`의
   Local First 정의는 "PC 저장소 / 외장 SSD / USB 저장소"라는 **파일
   기반** 저장을 예시로 든다. 현재 구현은 브라우저 관리 저장소다 —
   "로컬"이긴 하지만 사용자가 파일로 만지거나 USB로 옮길 수 있는
   형태가 아니다. 이 간극이 이 문서의 핵심 관찰이다.

## 2. 현 구조의 실제 위험

### 위험 1 — 브라우저 저장소 축출(eviction). 가장 크고, 유일하게 "지금" 아픈 위험

localStorage/IndexedDB는 브라우저 소유의 저장소라 브라우저가 지울 수
있다. 특히 주 사용 환경인 Safari/iPadOS는:

- (a) ITP(Intelligent Tracking Prevention)가 7일간 상호작용 없는
  사이트의 스크립트 쓰기 저장소를 삭제할 수 있다 — 홈 화면에 추가된
  웹앱은 예외로 알려져 있다. 로컬 IP 접속에 이 규칙이 적용되는지는
  **불명확하다(추측, 실측 필요)**.
- (b) 저장 공간 압박 시 IndexedDB(Media blob이 여기 있다)부터 축출될
  수 있다.

백업 기능이 없으므로 축출이 일어나면 **찬양 라이브러리(SongStore),
스타일 프리셋, Presentation 전체가 복구 불가로 유실**된다. 매주
반복되는 행사 직전에 이게 터지는 시나리오는 D-009(안정적 송출 우선)의
관점에서 최악이다. "2주 쉬고 왔더니 데이터가 없다"는 형태로 나타날
수 있어, 발생 전까지는 존재감이 없는 종류의 위험이다.

### 위험 2 — 통제 불가한 런타임

브라우저(iPadOS/Safari)는 자동 업데이트되고, 프로젝트는 그 위에서
동작한다. 9-9/9-10 같은 iOS 전용 렌더링 이슈가 OS 업데이트로 재발할
수 있다. 다만 이는 EXE로 바꿔도 WebView 기반(Tauri)이면 그대로 남고,
Electron이어야만 런타임이 고정된다 — 아래 선택지 비교 참조.

### 위험 3 — 다중 디스플레이/창 배치 운영 마찰

output.html은 별도 창/탭이라 매 행사마다 수동 배치가 필요하다.
브라우저의 창 자동 배치 API(Window Management)는 Chromium 전용이라
Safari에서는 불가. **다만 실사용 마찰 보고는 아직 없다** — 전체화면
버튼도 의도적으로 제거된 상태(UI/UX-2)로 문제없이 운영 중이다.

### 위험 4 — 로컬 서버 의존

"서버를 켠다"는 절차 자체가 운영 단계에 포함된다. 서버가 안 떠 있으면
앱 진입 자체가 안 된다(데이터는 브라우저에 있어도).

## 4. ~~판단 — EXE 전환 필요성~~ (2026-07-11 전제 정정으로 대체 — 8절 참조)

> 이 절의 결론은 "주 사용 환경 = iPad/Safari"라는 잘못된 전제 위에
> 서 있었다. 판단 과정 기록으로 원문을 보존하되, 유효한 결론은
> **8절 재평가**다. 흥미롭게도 이 절이 스스로 정의한 재검토 트리거
> 1번("주 운영 기기가 Windows/macOS PC로 이동")이 문서 작성 당일
> 발동된 셈이다.

**지금은 불필요하며, 현 사실관계에서는 방향 자체가 어긋난다.**

핵심 논거는 단순하다: **EXE는 Windows/macOS 해법인데 주 사용 환경이
iPad/Safari다.** EXE로 전환하면 주 환경을 버리거나(운영 기기 교체
강제), 두 배포 형태를 병행 유지해야 한다(단일 HTML 구조의 단순성
포기). EXE가 주는 세 가지 이점(저장 안정성/창 제어/런타임 고정) 중
실사용에서 지금 실제로 아픈 것은 **저장 안정성 하나뿐**이고, 그것은
선택지 B(내보내기/가져오기 + persist)로 EXE 없이, 무빌드 구조를
유지한 채, 훨씬 낮은 비용으로 해소된다.

또한 EXE 착수는 문서화된 선행 부채와 정면으로 만난다 — StorageAdapter
헤더와 TODO 9-18이 명시한 "AppStore 부팅이 동기(sync)라 실제 비동기
저장소 도입 시 부팅 순서 자체를 걷어내야 한다"는 한계가 그것이다.
즉 EXE는 "패키징 작업"이 아니라 "부팅 아키텍처 재설계 + 패키징 +
배포 채널 구축"이며, 지금 이 비용을 지불할 실사용 근거가 없다(YAGNI —
D-022/D-026과 같은 정신).

**권고 방향(Research 의견, 강제력 없음)**:

1. **단기 — B를 TODO 승격 후보로**: JSON 내보내기/가져오기(Presentation
   + Song + AppSettings, Media는 용량 문제로 범위 판단 필요) +
   `navigator.storage.persist()` 호출 + 홈 화면 추가 안내. 위험 1을
   직접 제거하는 최소 비용 조치이며 어떤 Decision과도 충돌하지 않는다.
2. **중기 — C(PWA)는 B 이후 별도 검토**: SW 캐시 무효화 전략이 배포
   흐름을 바꾸므로 독립 Research/Decision으로.
3. **장기 — D(EXE)는 트리거 대기**(아래).

## 8. 재평가 (2026-07-11 — 전제 정정: 실운영은 Windows/Mac 데스크톱·노트북 중심)

사용자 확인으로 운영 환경 전제가 바뀌어 4절 판단을 재평가한다.
초판 4절이 스스로 정의한 재검토 트리거 1번이 발동된 것이므로, 이
재평가 자체가 초판의 절차를 따르는 것이기도 하다.

### 8-1. 전제 변경이 바꾸는 것 / 안 바꾸는 것

**바뀌는 것**:

- 초판의 EXE 배제 핵심 논거("주 환경 iPad를 못 덮는다")가 소멸한다 —
  EXE는 이제 **정합적인 장기 종착지 후보**다.
- 운영 브라우저를 Chromium 계열(Chrome/Edge)로 표준화하는 선택지가
  열린다. 이것만으로 Window Management API(다중 디스플레이에 output
  자동 전체화면 — Safari 불가였던 것), File System Access API(파일
  기반 저장 — `Vocabulary.md` Local First 원의인 "PC 저장소/외장
  SSD/USB"와 정확히 합치), `storage.persist()`의 관대한 승인까지 —
  **웹 유지 전략이 도달할 수 있는 상한이 크게 올라간다.**
- PWA의 최대 이점이었던 "iPad 홈 화면 앱"이 의미를 잃는다 — PWA의
  남은 가치는 설치형 진입점 + 오프라인(위험 4 해소)으로 축소.
- 위험 1(저장소 축출)의 심각도가 내려간다 — 데스크톱 Chrome/Edge는
  Safari보다 축출 정책이 온건하고 persist() 승인이 잘 되는 것으로
  알려져 있다(세부는 착수 시 재확인). 단 "브라우저 프로필 정리/삭제 =
  데이터 소실" 구조 자체는 그대로이므로 **백업 기능의 필요성은
  불변**이다.

**안 바뀌는 것**:

- **B(내보내기/가져오기)가 최우선**이라는 결론 — 데이터 안전장치는
  패키징 방식과 무관하게 앱 자체의 기능이어야 한다. EXE로 가더라도
  내보내기는 필요하다(기기 이전/백업).
- EXE의 선행 부채 — AppStore 동기 부팅 해체(TODO 9-18)와 신규
  Decision 다수. 어떤 경로로 가든 EXE 착수 전 필수.
- "실사용 마찰 신호가 착수 근거"라는 원칙(D-022/D-026의 YAGNI 정신).

### 8-2. 세 전략 재평가

**웹 유지(A/B + Chromium 표준화) — 단기 최적, 그리고 생각보다 멀리 간다.**
B에 더해 운영 브라우저를 Chrome/Edge로 표준화하면(비용 0의 운영
결정), persist() 승인·창 자동 배치(Window Management)·파일 저장
확장(File System Access)까지 웹인 채로 확보 가능하다. 남는 구조적
약점: 로컬 서버 의존(위험 4), 런타임 자동 업데이트(위험 2), 그리고
"브라우저 앱"이라는 운영 형태 자체의 마찰(실수로 탭/창 닫기, 주소창
노출, 확대율 틀어짐).

**PWA — 데스크톱 전제에서는 어중간해짐.**
남는 이점(설치 아이콘, 독립 창, 오프라인)은 유효하나, "서버 의존
해소"가 목적이라면 EXE가 더 완전한 해법이라 이중 투자가 된다. SW
캐시 무효화가 "pull 즉시 반영" 배포 흐름을 바꾼다는 초판 지적도
그대로다. **기본 경로에서 제외**하되, EXE 착수 조건이 왔는데 선행
부채를 당장 지불할 수 없는 상황의 저비용 교량으로만 재고한다.

**EXE(Electron/Tauri) — 장기 종착지 후보로 승격. 단 "지금 당장"은 여전히 아니다.**
데스크톱 전제에서 EXE 고유 이점이 전부 살아난다: 더블클릭
실행(서버 완전 제거 — 위험 4의 근본 해소), OS 수준 창 제어(output을
두 번째 디스플레이에 자동 배치/키오스크), 런타임 고정(위험 2 해소),
파일 기반 저장 통합(Local First 원의 실현).

- **Electron vs Tauri**: D-009(송출 안정성) 관점에서 "런타임 고정"이
  EXE의 핵심 가치라면 **Electron이 논리적**이다 — Tauri는 OS가 자동
  업데이트하는 WebView(Windows는 WebView2/Edge 기반)에 의존하므로
  런타임 고정 이점이 사라지고, 그러면 "웹 유지 + Chromium 표준화"
  대비 차별점이 창 제어와 서버 제거만 남는다. Electron의 비용은 번들
  크기(100MB+)와 업데이트 채널 운영. (일반 지식 기반 — 버전별 세부는
  착수 시 재확인 필요.)
- **여전히 "지금"이 아닌 이유**: (1) 실사용 마찰 보고가 아직
  0건이다 — 창 배치도 서버 운영도 불편 신호가 TODO/CurrentState
  어디에도 없다. (2) 선행 부채(동기 부팅 해체)가 크다. (3) 최대
  위험(데이터 유실)은 B가 먼저, 훨씬 싸게 제거한다.

### 8-3. 수정된 권고 로드맵

1. **단기 — B 승격** (전제 변경과 무관하게 불변): JSON 내보내기/
   가져오기 + `storage.persist()`.
2. **단기~중기 — 운영 브라우저 Chromium 표준화** (운영 결정, 코드
   무수정). 창 배치 마찰이 실제로 확인되면 Window Management API를
   별도 항목으로 검토.
3. **장기 — EXE를 기본 종착지 후보로 유지.** 착수 조건(둘 중 하나):
   - (a) "브라우저라서 생기는" 운영 마찰(서버 켜기, 창 배치, 탭
     닫힘, 확대율)이 실사용에서 반복 확인됨
   - (b) 비기술 운영자에게 배포할 필요(더블클릭 실행)가 생김
   착수 시: Electron 우선 검토(D-009 근거) + AppStore 비동기 부팅
   선행 + 부팅/저장/배포 신규 Decision 작성.
4. PWA는 기본 경로에서 제외(8-2 참조).

초판 4절의 트리거 목록은 이 절의 착수 조건으로 대체된다.
