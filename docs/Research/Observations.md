### ProPresenter는 왜 Playlist(Flow)와 Library를 따로 둘까?

가설: 재사용 가능한 콘텐츠(Asset/Song)와 "이번 행사에서 실제로 쓸 순서
(Playlist)"는 라이프사이클이 다르다 — Asset은 여러 행사에 걸쳐 계속
남아있고, Playlist는 행사 단위로 매번 새로 짜여진다. 하나의 flat
리스트로 두 개념을 같이 다루려 하면, "순서를 편집하는 것"과 "무엇을 쓸지
고르는 것"이라는 서로 다른 조작이 같은 화면 하나에 눌려 담기게 된다.

확인이 더 필요한 부분: 이 프로젝트 규모(단일 행사 중심, MVP)에서 이
분리가 정말 필요한 시점이 언제인가? Song/Asset 재사용이 실제로 발생하기
시작하는 시점(TODO.md의 "Asset/Song 관계 재검토" 항목과 연결)이 아마
이 질문에 답할 트리거일 것 같다 — 지금 당장은 아니다.

### 열린 질문: Section이 진짜 "Editor 개념"으로 남아도 되는가

`FutureEditor.md`는 Section을 "Page를 그룹화하는 Editor 개념(송출 단위
아님)"으로 정의했다. Page 중심 모델(D-Editor-4)로 바뀌어도 이 정의

아니면 Library 쪽 개념(예: Playlist)과 관계를 맺어야 하는지는 다시 물어야
할 것 같다.

## 2026-07-05

### ProPresenter 상단 메뉴바 탐색 (Workflow 관찰)

Flow/Browser 분리 다음으로, ProPresenter가 화면을 어떻게 더 잘게
나누는지를 상단 메뉴바 기준으로 훑어봤다. 네 가지가 눈에 띄었다.

**1. Show와 편집 — 가장 큰 단위의 분화.** 운영(Show)은 이미 작성/세팅
완료된 콘텐츠를 그대로 제공해야 하는 모드라, 편집 모드와 분리해서
"변동성이 낮은" 상태로 만들어둔다. Show 화면에서는 콘텐츠가 실수로
바뀌지 않는다는 확신이 중요해 보인다.

**2. Reflow(리플로우) 기능.** Show 모드 중에도 돌발 상황에는 대응해야
하니 만든 긴급 수정 기능으로 보인다. 다만 상시로 켜져 있는 게 아니라
상단 메뉴에서 따로 활성화해야 접근 가능하고, 편집 가능한 대상도 텍스트로
한정돼 있다 — "웬만하면 안 건드리게" 만들어 놓고, 정말 필요할 때만
좁은 범위로 열어주는 설계로 읽힌다.

**3. 성경 등 확장 기능.** 사용자가 뭘 하는 사람인지(교회? 강의? 행사?)
미리 알 수 없으니, 커스텀해서 쓸 수 있게 확장 지점을 열어둔 것으로
보인다.

**4. 텍스트(글꼴/Bold/대소문자/정렬/스트로크/그림자 등)와 테마, 검색.**
"편집 모드에 들어갔을 때의 스타일링과 Show 중 미리보기의 스타일링이 왜
다른가"라는 의문이 있었는데, 이렇게 정리됐다: 라이브러리에 텍스트와
편집 내용을 미리 저장해뒀어도, 그때그때 테마를 바꿔서 작업 시간을
줄이거나 다른 느낌을 낼 수 있게 하려는 것 같다. 텍스트 편집 기능
자체는 "편집 모드가 미처 상정하지 못한 돌발 상황"에 대응하기 위한 여지로
보인다. 검색도 비슷한 맥락 — 사전에 준비는 돼 있었지만 운용 중 갑자기
필요해진 텍스트를 바로 적용시키기 위한 도구.

### 결론

이 네 가지를 관통하는 인상은, ProPresenter가 "사용자가 원할 수 있는
기능"과 "사용자가 피하고 싶은(실수하고 싶지 않은) 상황" 양쪽을 모두
Workflow 설계에 반영하고 있다는 것이다 — Show는 안정성을, Reflow/검색은
유연성을 각각 좁은 범위로 담당하게 나눠놓은 느낌.

이 프리젠터도 같은 질문을 가져가야 할 것 같다: 사용자가 쓰고 싶어하는
기능과, 운영 중 마주칠 수 있는 돌발 상황 각각을 어떤 화면/모드에
배치할지를 Flow/Browser 분리보다 더 세부적인 층위에서 계속 관찰할 필요가
있다. (Flow와 Browser 사이의 "Show vs Edit" 구분은 이미 이 프로젝트의
`presenterState.appMode`(edit/live)와 개념적으로 겹치는 부분이 있어
보인다 — 다만 지금은 우연히 이름이 비슷한 정도이고, Reflow 같은 "제한된
긴급 수정" 개념은 아직 이 프로젝트에 없다.)

### Reflow 기능 실제 조사 (위 2번 추측 정정, 2026-07-05)

위 "2. Reflow" 항목은 실제 사용해보지 않고 이름과 정황만으로 "긴급 수정
기능"이라 추측한 것이었다. Claude가 웹 검색으로 실제 동작을 확인한 결과,
정확히는 다음과 같다 — 추측이 틀렸으므로 정정해둔다:

- **핵심은 "긴급 대응"이 아니라 "이미 여러 슬라이드로 쪼개진 곡 가사를,
  다시 통짜 텍스트로 보면서 재편집"하는 도구다.** 왼쪽에 곡 전체
  텍스트, 오른쪽에 슬라이드 미리보기를 같이 띄워두고, 줄바꿈/슬라이드
  나누기 지점을 텍스트 편집하듯 조정하면 슬라이드가 실시간으로
  다시 나뉜다.
- 그룹 이름/색상(우리 프로젝트의 Section 제목/색상과 비슷한 개념)도 이
  화면에서 같이 붙일 수 있다.
- 이미 Show/Live에 올라간 곡이라도 실시간으로 재구성 가능(공식 문서
  표현: "editing the slides in real-time").
- 슬라이드당 텍스트박스 하나인 단순 구조에서만 동작한다 — 텍스트박스
  여러 개인 슬라이드는 지원 안 됨.

**우리 프로젝트에 왜 지금은 안 되는가**: `utils/lyricsImport.js`의 기존
주석이 정확히 이 문제를 미리 짚어두고 있었다 — "실행 후 rawText는
버려진다. Page[]만 남는다." `importLyrics()`는 빈 줄 기준으로 잘라
`createTextPage()`를 만들고 끝나는 **일방향(one-way) 파이프라인**이라,
Page가 만들어지는 순간 원본 텍스트와의 연결이 끊긴다. Reflow는 본질적으로
Page들과 원본 텍스트를 양방향으로 오갈 수 있어야 하므로, 지금 구조 위에
그대로 얹을 수 없다.

**필요한 최소 변경(구현하면)**: 연속된 Text Page 여러 개의 `text`를
다시 이어붙여 편집창에 띄우고 → 사용자가 줄바꿈 수정 → 같은 분절
로직으로 재분절 → 그 범위의 기존 Page들을 새 Page들로 통째로 교체(원래
위치와 `sectionId`를 물려받도록). Reducer에 배치 교체용 액션 하나
(`REPLACE_PAGE_RANGE` 류) 추가 정도로 기술적으로는 크게 어렵지 않다.

**다만 걸리는 것 두 가지**:
1. **Live 안전성**: 재분절하면 Page id가 전부 새로 생기는데,
   `presenterState.livePageId`는 특정 id를 참조한다 — 지금 송출 중인
   Page가 그 범위에 포함되면 참조가 끊긴다. 최소한 "Live 중인 Page가
   포함된 범위는 재분절 금지" 정도의 안전장치가 필요해 보인다.
2. **Song Aggregate 문제와 직결**: Reflow를 제대로 만들려면 "가사
   원본 텍스트를 유지하는 개념"이 필요한데, 이게 정확히
   `TODO.md`의 "Asset/Song 관계 재검토"(Song이 Asset 하나인지, 아니면
   Page 여러 개 + 메타데이터를 묶는 별도 Aggregate인지)가 풀어야 할
   질문과 같다. Song Aggregate 없이 Reflow만 먼저 만들면, 나중에 Song
   Aggregate를 도입할 때 다시 손봐야 할 가능성이 높다.

**결론 (Claude 의견)**: 지금 바로 만들 우선순위는 아니라고 본다. Section
때와 같은 패턴 — "있으면 좋을 것 같다"는 근거는 있지만, 아직
"실사용 중 이게 없어서 불편했다"는 신호가 CurrentState.md 어디에도 없다.
Song Aggregate 논의가 실제로 시작될 때(TODO.md "Asset/Song 관계 재검토"
착수 시점) Reflow를 그 설계 안에 자연스럽게 끼워 넣는 게 순서상 맞다.

## 2026-07-08

### Research: Live Emergency Overlay System

**배경.** 예배나 발표 진행 중에는 예상하지 못한 돌발 상황이 발생할 수 있다 — 영상 재생 오류, 음향 점검, 온라인 송출 장애, 행사 시작 지연, 잠시 대기 안내 같은 경우다. 이런 상황에서는 현재 송출 중인 화면을 유지한 채로, 운영자가 매우 짧은 시간 안에 안내 문구를 송출할 수 있어야 한다.

**목표.** 운영자가 **사전 교육 없이도** 수 초 내에 긴급 안내를 송출할 수 있는 Overlay 시스템을 제공한다. 이 기능은 일반 슬라이드 작성 기능과 명확히 구분되는 **운영(Operational) 기능**으로 정의한다.

**핵심 요구사항**은 다음 7가지다.

1. **즉시 접근 가능** — 메뉴를 여러 단계 탐색하지 않아도 되고, Live 운용 화면에서 항상 접근 가능해야 하며, 운영자가 기능 위치를 미리 외우지 않아도 쓸 수 있어야 한다. 목표치는 "긴급 상황 발생 후 수 초 이내 메시지 송출".
2. **일반 슬라이드와 완전히 분리** — 긴급 메시지는 Presentation(Page)이 아니다. `Presentation`(Page/Background/Cue)과 별개로 `Overlay`(Emergency Message)라는 독립된 State로 관리해, 프로젝트 저장 데이터와 운영 상태를 명확히 나눈다.
3. **현재 송출 화면 유지** — 긴급 메시지는 슬라이드를 바꾸지 않고, Live Page도 바꾸지 않고, 프로젝트 자체도 수정하지 않는다. 현재 송출 화면 위에 Overlay만 추가로 렌더링하며, Overlay를 제거하면 원래 화면이 즉시 복원되어야 한다.
4. **렌더 우선순위** — Overlay는 항상 최상단에서 렌더링한다. 순서는 Background → Video → Page Content → Cue → Emergency Overlay.
5. **메시지 입력** — 운영자는 "메시지 입력 → 송출" 두 단계만으로 사용할 수 있어야 한다. 복잡한 설정은 지양한다.
6. **위치 선택** — 메시지 위치(상단/중앙/하단)를 빠르게 선택 가능해야 한다. 세밀한 좌표 조정보다는 운영 속도를 우선한다.
7. **출력 대상 선택** — Overlay는 출력(Display)별로 독립적으로 송출 가능해야 한다(예: 메인 스크린·LED엔 켜고, Confidence Monitor·온라인 송출·OBS 출력엔 끄는 식). 현장 안내와 온라인 안내가 서로 다른 경우를 고려한다.

**UX 원칙.** 운영자는 긴급 상황에서도 생각하지 않아야 한다 — 기능은 항상 눈에 띄어야 하고, 클릭 수가 최소여야 하며, 현재 Live 운용을 방해하지 않아야 한다.

**향후 확장 가능성.** Emergency Overlay는 향후 Overlay Engine의 첫 번째 기능으로 쓸 수 있다. 같은 시스템으로 Countdown, QR Code, 온라인 전용 안내, 방송 전용 자막, 현장 전용 공지, 임시 안내 배너 같은 기능도 지원할 수 있도록 설계한다.

**MVP 범위.** 2차 MVP는 텍스트 Overlay, 상/중/하 위치, 즉시 송출, 즉시 종료, 출력 대상 선택까지만 지원한다. 애니메이션, 스타일 편집, 템플릿 관리, 이미지/동영상 Overlay는 제외한다. 핵심 목표는 "운영자가 실제 예배 중 수 초 내에 긴급 안내를 송출할 수 있는 것"이다.
